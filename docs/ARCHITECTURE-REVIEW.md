## 一、架构优势

### 1.1 清晰的分层设计 ✅

```
创意层（~scriptwriter）
    ↓
本体层（~build-ontology）✨ 新增
    ↓
资产层（~asset-factory）✨ 新增
    ↓
生成层（~batch/~start）
    ↓
质检层（Phase 6: Audit & Repair）✨ 新增
```

**优势**:
- 每层职责单一，边界清晰
- 上层输出是下层输入，数据流顺畅
- 可独立测试、独立优化

### 1.2 状态驱动架构 ✅

**核心设计**:
- 每个 shot 有完整的状态快照（shot packet）
- 状态包含：角色状态、资产引用、本体约束、禁止变化项
- 状态可追溯、可验证、可修复

**优势**:
- 可重现：知道每个镜次用了哪些输入
- 可调试：失败时知道哪里出错
- 可修复：知道如何调整参数

### 1.3 资产复用机制 ✅

**设计**:
- 角色定妆包：一次生成，跨集复用
- 场景 styleframe：一次生成，跨项目复用
- 道具包：按需生成，状态追踪

**优势**:
- 成本降低：不需要每个镜次都生图
- 一致性提升：同一角色在 720 个镜次中外貌一致
- 可扩展：新项目可直接复用已有资产

### 1.4 自动质检与修复 ✅

**设计**:
- 3 层 QA：Symbolic（硬逻辑）+ Visual（画面）+ Semantic（戏剧）
- 3 种修复策略：pass / local_repair / regenerate
- 自动决策：根据 severity 自动选择修复策略

**优势**:
- 降低人工成本：不需要逐个检查 720 个视频
- 提升质量：自动发现人工容易忽略的问题
- 降低返工率：30% → 5%（预期）

### 1.5 向后兼容 ✅

**设计**:
- 所有新 phases 都是可选的
- 系统自动检测并回退到旧流程
- 不影响现有功能

**优势**:
- 平滑升级：可以逐步启用新功能
- 风险可控：新功能失败不影响旧流程
- 灵活性高：根据项目需求选择流程

## 二、潜在风险与建议

### 2.1 复杂度风险 ⚠️

**问题**:
- v2.0 新增 4 个 phases（0, 2.5, 3.5, 6）
- 新增 7 个 agents（ontology-builder, asset-factory, memory, shot-compiler, qa, repair）
- 流程变长：Phase 1-5 → Phase 0-6

**影响**:
- 调试难度增加
- 失败点增多
- 运行时间变长

**建议**:
1. **分阶段启用**：
   - 第一阶段：只启用 Phase 0 + Phase 2.5（本体 + 资产）
   - 第二阶段：启用 Phase 3.5（Shot Packet）
   - 第三阶段：启用 Phase 6（Audit & Repair）

2. **增强可观测性**：
   - ✅ 已有 Trace 系统（`~trace`）
   - 建议：为每个新 phase 添加详细的 trace 日志
   - 建议：添加 phase 级别的性能监控

3. **提供降级开关**：
   ```yaml
   # config/features.yaml
   enable_ontology: true
   enable_asset_factory: true
   enable_shot_packet: true
   enable_audit_repair: true
   ```

### 2.2 性能风险 ⚠️

**问题**:
- Phase 0: 每集需要 LLM 调用提取本体（~1-2 分钟/集）
- Phase 2.5: 生成资产包需要大量 API 调用（~5-10 分钟）
- Phase 3.5: 每个 shot 需要编译 packet（~5-10 秒/shot）
- Phase 6: 每个 shot 需要 QA + 可能的修复（~10-30 秒/shot）

**影响**:
- 10 集剧本：Phase 0 + 2.5 约 20-30 分钟
- 60 集剧本：Phase 0 + 2.5 约 2-3 小时
- 每集 12 个 shot：Phase 3.5 + 6 约 3-8 分钟/集

**建议**:
1. **并行化**：
   - ✅ Phase 0 可并行（每集独立）
   - ✅ Phase 2.5 可并行（每个资产独立）
   - ⚠️ Phase 3.5 需串行（依赖前一镜）→ 建议优化为并行
   - ✅ Phase 6 可并行（每个 shot 独立）

2. **缓存机制**：
   - ✅ 已有幂等性检查（资产存在则跳过）
   - 建议：为 Phase 0 添加缓存（相同剧本不重复提取）
   - 建议：为 Phase 3.5 添加缓存（相同 shot 定义不重复编译）

3. **增量更新**：
   - 建议：`~build-ontology --incremental`（只处理新增/修改的剧本）
   - 建议：`~asset-factory --incremental`（只生成缺失的资产）

### 2.3 成本风险 ⚠️

**问题**:
- Phase 0: LLM 调用成本（每集 ~$0.01-0.05）
- Phase 2.5: 图像生成成本（每个资产 ~$0.02-0.10）
- Phase 6: 额外的 LLM 调用 + 可能的重新生成

**影响**:
- 10 集剧本：Phase 0 + 2.5 约 $5-20
- 60 集剧本：Phase 0 + 2.5 约 $30-120
- Phase 6 修复：每个失败 shot 额外 $0.50-2.00

**建议**:
1. **成本监控**：
   ```bash
   ~cost-report          # 查看总成本
   ~cost-report ep01     # 查看单集成本
   ~cost-report --phase  # 按 phase 分组
   ```

2. **成本优化**：
   - 使用更便宜的模型（Phase 0 可用 Haiku）
   - 批量调用（减少 API 调用次数）
   - 智能跳过（低风险 shot 跳过 Phase 6）

3. **预算控制**：
   ```yaml
   # config/budget.yaml
   max_cost_per_episode: 10.00  # 单集最大成本
   max_cost_per_project: 600.00 # 项目最大成本
   alert_threshold: 0.8         # 80% 时告警
   ```

### 2.4 数据一致性风险 ⚠️

**问题**:
- 多个数据源：剧本、角色档案、场景档案、本体模型、资产包
- 数据可能不同步：修改剧本后，本体模型和资产包未更新

**影响**:
- 生成的视频与剧本不一致
- QA 检测到大量错误
- 需要手动清理和重新生成

**建议**:
1. **版本控制**：
   ```json
   {
     "script_version": "v1.2",
     "ontology_version": "v1.2",
     "assets_version": "v1.1",
     "last_sync": "2026-04-01T10:00:00Z"
   }
   ```

2. **依赖追踪**：
   ```bash
   ~check-sync          # 检查数据是否同步
   ~sync --force        # 强制重新同步
   ```

3. **自动更新**：
   - 修改剧本时，自动标记本体模型为 stale
   - 修改角色档案时，自动标记资产包为 stale
   - 运行 ~batch 前，自动检查并更新 stale 数据

### 2.5 错误传播风险 ⚠️

**问题**:
- Phase 0 的错误会传播到 Phase 2.5、3.5、6
- 例如：本体模型中角色变体错误 → 资产工厂生成错误的图 → 所有镜次都用错误的图

**影响**:
- 大规模返工
- 浪费 API 调用成本
- 延长项目周期

**建议**:
1. **早期验证**：
   - Phase 0 完成后，立即验证本体模型
   - Phase 2.5 完成后，立即验证资产包（人工审核主角）
   - 不通过则阻断后续流程

2. **检查点机制**：
   ```bash
   ~validate-ontology ep01    # 验证本体模型
   ~validate-assets           # 验证资产包
   ~validate-shot-packet ep01 shot-03  # 验证 shot packet
   ```

3. **回滚机制**：
   ```bash
   ~rollback ep01 --to phase2  # 回滚到 Phase 2
   ~rollback --all --to phase0 # 全部回滚到 Phase 0
   ```

## 六、总结

1. **架构清晰**: 分层设计，职责单一
